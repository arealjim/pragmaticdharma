## 1. Project summary

Radio is a **self-hosted music streaming stack** running on `devbox` and exposed publicly via a Cloudflare Tunnel at `radio.pragmaticdharma.org`.

**Stack**
- **Navidrome** (Docker) — Subsonic-compatible music server + web UI, bound to `127.0.0.1:4533`
- **Gluetun + Transmission** (Docker) — torrent client routed through a ProtonVPN WireGuard tunnel with kill switch, bound to `127.0.0.1:9091` via gluetun
- **Auth proxy** (Node.js / Express 4 + `http-proxy-middleware` 3) — listens on `127.0.0.1:4534`, runs as `radmin` via `radio-proxy.service` systemd unit
- **yt-dlp** (spawned subprocess) — powers YouTube search, radio (YouTube Mix), and background audio downloads
- **Cloudflare Tunnel** (`cloudflared`) — fronts `radio.pragmaticdharma.org` → `127.0.0.1:4534`

**Auth style:** worker-gate via the Node auth proxy. For web requests, it validates the `pd_session` JWT cookie (HMAC-SHA256, ported to Node `crypto` from `shared/auth-cloudflare.js`), checks `hasRadioAccess` (backward-compat: no `projects` claim = allow), and sets a `Remote-User` header before proxying to Navidrome. For `/rest/*` (Subsonic API), the middleware **skips JWT auth entirely** — Subsonic clients are expected to use the Navidrome token+salt scheme.

**Claude usage:** **None.** `yt-dlp`, Navidrome's Subsonic API, and Last.fm are the only external calls. Not a Claude cost concern.

**Media specifics:** Opus audio files written to `data/music/<artist>/<album>/<track>.opus`, scanned by Navidrome. No URL signing; no separate CDN; audio streams flow Cloudflare Tunnel → proxy → Navidrome. Torrents are network-isolated inside the gluetun container (VPN kill switch via `FIREWALL_VPN_INPUT_PORTS=9091`).

## 3. Critical findings

### C1. Subsonic API (`/rest/*`) accepts spoofed `Remote-User`, allowing authentication bypass

**Location:** `src/proxy/index.js:31-48` and `docker-compose.yml:50-51`

The auth middleware skips JWT verification for `/rest/*`:
```js
app.use((req, res, next) => {
  if (req.path.startsWith('/rest')) return next();
  // ... JWT check for everything else
});
```

Navidrome is configured as a trusting reverse-proxy client:
```yaml
ND_REVERSEPROXYUSERHEADER: Remote-User
ND_REVERSEPROXYWHITELIST: 172.16.0.0/12,127.0.0.0/8
```

The whitelist is IP-based and applies to the connection Navidrome sees — which is the auth-proxy at `127.0.0.1`, so the whitelist **always passes**. Because the middleware returns early on `/rest/*` without scrubbing headers, any client can send:

```
GET /rest/getArtists?u=admin&... HTTP/1.1
Cookie: (none)
Remote-User: admin
```

and `http-proxy-middleware` will forward that `Remote-User` verbatim to Navidrome, which honors it as an authenticated reverse-proxy login. Navidrome has documented behavior of treating `ND_REVERSEPROXYUSERHEADER` as proof of identity for every request (including Subsonic), bypassing the password+salt+token dance.

**Impact:** Unauthenticated remote attacker can enumerate the library, stream any song, and — depending on Navidrome's role model for the reverse-proxy user — potentially upload/edit playlists or mutate state. Completely circumvents the `radio` project gate.

**Fix:**
1. In the auth middleware, **always** delete incoming `remote-user` (lowercase) before branching on `/rest/*`:
   ```js
   delete req.headers['remote-user'];
   ```
2. For `/rest/*`, either (a) still require a valid `pd_session` JWT and fall through to Subsonic's own password auth, or (b) keep the bypass but ensure no Remote-User is attached. Option (a) is strictly better — Subsonic clients can carry cookies.
3. Consider switching Navidrome to plain password auth and dropping `ND_REVERSEPROXYUSERHEADER` entirely; the proxy adds little over Navidrome's native auth once the platform JWT gate is enforced.

### C2. `Remote-User` is set on the wrong case, so the proxy never forwards a trusted identity

**Location:** `src/proxy/index.js:46`

```js
req.headers['Remote-User'] = payload.sub || payload.email || 'admin';
```

Node.js normalizes incoming headers to **lowercase** in `req.headers`. Assigning the mixed-case key `Remote-User` creates a new sibling property that coexists with any lowercase `remote-user` the client sent. Whether `http-proxy-middleware` forwards the intended value depends on its internal header-serialization logic, but in practice:

- If a client sets `remote-user: attacker`, it sits in `req.headers['remote-user']` and is forwarded as-is on `/rest/*` (see C1).
- On authenticated non-`/rest` routes, the Mixed-Case key is sometimes dropped during proxying, so Navidrome sees no Remote-User and falls back to anonymous, or — worse — sees the **attacker-supplied** `remote-user` from the request.

**Impact:** Identity spoofing. A legitimate user's cookie is validated, but the identity Navidrome records may be whatever the client put in `remote-user`. Combined with C1, lets an attacker choose any identity (e.g., `admin`).

**Fix:**
1. Always `delete req.headers['remote-user']` at the top of the middleware.
2. Set the trusted header with lowercase key: `req.headers['remote-user'] = payload.sub || ...`.
3. Add a unit/integration test that sends `Remote-User: evil` and asserts Navidrome sees the JWT subject, not `evil`.

## 4. High findings

### H1. Auth proxy accepts arbitrary YouTube video IDs from request body, enabling cost/bandwidth abuse by any approved user

**Location:** `src/api/download.js:180-208`

```js
const { id, title } = req.body || {};
if (!id || !title) return res.status(400).json({ error: 'id and title required' });
// ...
const url = `https://www.youtube.com/watch?v=${d.id}`;
```

`id` is not validated against the YouTube 11-char video-ID format. Any string is interpolated into a URL that yt-dlp fetches. yt-dlp accepts many URL shapes and extractors, so creative `id` values can make the server download:

- Long movies or TV rips (several GB each) — consuming disk, egress quota, and ProtonVPN bandwidth.
- Live streams (indefinite download).
- URLs that exfiltrate via side-channels (yt-dlp will resolve redirects, hit SSRF-interesting targets if the id is crafted; though the URL template prefixes `https://www.youtube.com/watch?v=`, a value like `foo&url=http://169.254.169.254/...` still stays on the youtube.com host).

Queue cap is `MAX_QUEUED=20` with `MAX_CONCURRENT=2`, but there is no per-user or global rate limit across time — a user can drip-feed downloads indefinitely.

**Impact:** Approved-but-malicious platform user (or an attacker who steals any `pd_session`) can burn devbox disk, VPN bandwidth, and get the server's VPN egress IP flagged or banned by YouTube.

**Fix:**
- Validate `id` with `/^[A-Za-z0-9_-]{11}$/` before constructing the URL.
- Add a max duration check (from the metadata phase) and abort if > N minutes.
- Add a per-user daily download count limit (e.g., 50/day) backed by SQLite or a simple JSON file.
- Consider checking yt-dlp `filesize_approx` against a ceiling.

### H2. Search/radio endpoints spawn yt-dlp with no rate limit — DoS risk

**Location:** `src/api/search.js:6`, `src/api/radio.js:74,109`

Each request spawns a new `yt-dlp` child process (15–30 s timeout). There is no concurrency cap, no debouncing, and no per-user rate limit. An authenticated user (or a client with a stolen cookie) can burst N requests and exhaust CPU/memory on devbox.

**Impact:** Local DoS; yt-dlp forks hold memory for the duration of the timeout. Combined with `downloads.events` SSE connections (also unbounded), the process can be killed by the OOM killer.

**Fix:**
- Add a small rate limiter (e.g., `express-rate-limit` or a hand-rolled Map-based counter) keyed on `payload.sub`: 10 search/min, 5 radio/min.
- Cap concurrent yt-dlp processes across search+radio+download (global semaphore).

### H3. yt-dlp `stderr` is echoed back to the client on failure, leaking filesystem/internal paths

**Location:** `src/api/search.js:44`, `src/api/download.js:153`, `src/api/radio.js:26,104,146`

```js
return res.status(502).json({ error: 'Search failed', detail: stderr.slice(0, 200) });
```

yt-dlp error output frequently contains absolute paths (`/home/radmin/...`), Python tracebacks, IP addresses (especially the VPN egress), and cookie-handling warnings. The 200-byte truncation is tiny but still leaks.

**Impact:** Information disclosure to any authenticated user (or via XSS on other sites that embed a fetch).

**Fix:** Log stderr server-side; return generic `{ error: 'upstream failure' }` with an opaque request ID the admin can grep for.

### H4. Subsonic/Navidrome credentials in `.env` are readable by any process running as `radmin`

**Location:** `src/api/radio.js:50-71`, `/etc/radio/env`

`NAVIDROME_USER` / `NAVIDROME_PASSWORD` live in the systemd `EnvironmentFile=/etc/radio/env`. The Subsonic token scheme is `md5(password + salt)`, so leakage of the plaintext password lets an attacker compute any token. Permissions on `/etc/radio/env` should be verified (should be `0600 root:root` or `0640 root:radmin`, not world-readable).

Additionally, the salt is only 8 alphanumeric lowercase chars (`Math.random().toString(36).slice(2, 10)`) and seeded by `Math.random()` — not cryptographically random. For Subsonic this is mostly fine (Navidrome verifies `md5(pass+salt)`), but a low-entropy salt makes precomputed-table attacks marginally easier if tokens are ever intercepted.

**Fix:**
- `chmod 600 /etc/radio/env`, owner `root:radmin`, mode `0640`.
- Replace `Math.random()` salt with `crypto.randomBytes(8).toString('hex')`.
- Long term: use Navidrome's API key / personal access token feature instead of storing the user password.

## 5. Medium findings

### M1. `MUSIC_DIR` uses relative `resolve('data/music')` — CWD-dependent

**Location:** `src/api/download.js:7`

`resolve('data/music')` resolves against `process.cwd()`, not the module path. If the service is ever started from a different CWD (manual `node src/proxy/index.js` from `/tmp`, for example), downloads land in `/tmp/data/music/...` — effectively writing into an arbitrary directory as `radmin`.

**Fix:** `resolve(dirname(fileURLToPath(import.meta.url)), '../../data/music')` — same pattern already used in `proxy/index.js` for the static dir.

### M2. Sanitizer allows path-traversal-adjacent characters in `artist`/`album`/`track`

**Location:** `src/api/download.js:81`

```js
const sanitize = (s) => s.replace(/[<>:"/\\|?*]/g, '_').replace(/\.+$/, '').trim();
```

Removes `/ \ < > : " | ? *` and trailing dots. Does **not** strip:
- Leading `.` / `..` — an artist named `..` yields `MUSIC_DIR/../album/track.opus`, writing outside the music tree.
- Null bytes — JavaScript strings permit `\x00`; yt-dlp may truncate filenames at null, but Navidrome and the FS layer differ.
- Control chars and Unicode RTL overrides.

Metadata comes from YouTube — untrusted — and is used verbatim in `-o` template with literal `..`.

**Impact:** An attacker queuing a YouTube video whose uploader name is `..` could overwrite files one directory up (still inside `data/` in the normal deploy, but potentially outside `music/`).

**Fix:** After sanitization, explicitly reject `.` and `..`, and use `path.join(MUSIC_DIR, artist, album, …)` + check the result `startsWith(MUSIC_DIR + sep)`.

### M3. No CSRF protection on `POST /api/download` and `DELETE /api/download/:id`

The proxy uses cookie auth. Any page a user visits in the same browser can issue `fetch('https://radio.pragmaticdharma.org/api/download', {method:'POST', credentials:'include', ...})` and — if CORS is permissive, or the attacker gets the user to follow a form-post — trigger downloads on the victim's behalf.

Whether this is exploitable depends on how the platform sets `pd_session`. If it's `SameSite=Lax` (the usual default) most cross-origin POSTs are blocked; `SameSite=None` would be dangerous. Verify in `pragmaticdharma/worker.js` JWT cookie code.

**Fix:** Require a custom header (`X-Requested-With: fetch`) on state-changing routes; same-origin fetch is allowed to set it but cross-origin form posts cannot.

### M4. XSS via YouTube thumbnail id — `track.thumbnail` interpolated into `img src` without escaping

**Location:** `src/web/app.js:28-36`

```js
card.innerHTML = `
  <img src="${track.thumbnail}" alt="" ...>
  ...
  <button class="btn-dl..." data-id="${track.id}" ...`;
```

`track.thumbnail` is server-constructed as `https://i.ytimg.com/vi/${item.id}/mqdefault.jpg` where `item.id` is yt-dlp output. `track.id` is also interpolated into `data-id="${track.id}"`. All other fields go through `esc()`. A YouTube video ID is conventionally `[A-Za-z0-9_-]{11}`, but yt-dlp's JSON `id` field is not strictly guaranteed; a crafted playlist entry could contain `"`.

**Impact:** Low in practice (YouTube sanitizes its own IDs), but an XSS in this context steals the `pd_session` cookie → platform-wide takeover. Defense-in-depth is warranted.

**Fix:** Use `esc()` on `track.id` and construct the `<img>` via `document.createElement`/`img.src = ...` assignment instead of `innerHTML`, or add a strict `Content-Security-Policy` response header (none is currently set).

### M5. No CSP / security headers on proxy responses

The Express app sets no `Content-Security-Policy`, `X-Content-Type-Options`, `Referrer-Policy`, or `Strict-Transport-Security`. HSTS is typically added by Cloudflare, but CSP and XCTO must come from the origin.

**Fix:** Add `helmet` or hand-roll a small middleware:
```js
app.use((req, res, next) => {
  res.setHeader('Content-Security-Policy', "default-src 'self' https://i.ytimg.com; img-src 'self' https://i.ytimg.com data:; script-src 'self'; style-src 'self' 'unsafe-inline'");
  res.setHeader('X-Content-Type-Options', 'nosniff');
  res.setHeader('Referrer-Policy', 'same-origin');
  next();
});
```

### M6. SSE endpoint has no per-connection limit and keeps `res` refs forever on error

**Location:** `src/api/download.js:223-244`

`sseClients.add(res)` but cleanup relies on `req.on('close')`. If a client connects and the server-side `res.write` throws (pipe closed), the error isn't caught and the set never loses that entry. Over time, `broadcast()` calls grow slower; memory leaks.

**Fix:** Wrap `res.write` in try/catch and remove the client on any write error. Cap `sseClients.size` (e.g., 100) and reject further connections.

## 9. Summary

The **two critical findings are intertwined** and urgent:

1. The `Remote-User` header is not scrubbed on incoming requests, so anyone hitting `/rest/*` can present `Remote-User: admin` and — because Navidrome trusts any reverse-proxy connection from `127.0.0.0/8` — authenticate as any Navidrome user, completely bypassing the platform's `radio` project gate.
2. The same header is also set with mixed-case in the middleware, so the proxy's intended identity is never reliably forwarded for legitimate users either.

Fix: delete `req.headers['remote-user']` at the top of the auth middleware, set it with a lowercase key, and require a valid `pd_session` JWT on `/rest/*` as well (Subsonic clients in a browser carry cookies; dedicated clients like Symfonium can be configured to present the cookie, or can use Subsonic's own password auth with a separate Navidrome account that is NOT the reverse-proxy user).

Beyond the criticals, the project needs input validation on YouTube video IDs (H1), rate limiting on yt-dlp spawns (H2), path-traversal hardening on downloaded-file sanitization (M2), and a small dose of security headers (M5). No Claude cost concerns.
